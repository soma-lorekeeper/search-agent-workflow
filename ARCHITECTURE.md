# 아키텍처 상세

`readme.md`가 빠른 시작용이라면, 이 문서는 "왜 이렇게 만들었는지"와 "실제로 어떻게
동작하는지"를 자세히 설명한다. 전체 그림을 파악하는 용도.

## 1. 무엇을 만들었나

『전지적 독자 시점』 1~6화 원문을 인덱싱해 두고, 그 내용에 대해 자유롭게 질문하면
정확하게(근거 없는 내용을 지어내지 않고) 답하는 채팅 에이전트. 목표는 "이 7개 질문에만
답하는 봇"이 아니라 **범용 질의응답 에이전트**를 만드는 것 — 실제로 이후 held-out
질문, 부정 케이스, 멀티턴, 전역 요약 질문 등으로 계속 검증하며 특정 질문에 맞춘 하드코딩을
피했다.

## 2. 저장소 계층 — 왜 DB를 3개나 쓰나

| DB | 용도 | 왜 |
|---|---|---|
| **Qdrant** (VectorDB) | 원문 청크 임베딩 | 줄거리 회상·서술 묘사처럼 "정확한 키워드가 아니라 의미로 찾아야 하는" 질문에 강하다. |
| **Neo4j** (GraphDB) | 엔티티/관계 사실 | 소유·관계·사건처럼 구조화된 사실, 특히 "아이템이 여러 인물을 거쳐 이동" 같은 **다단계 연쇄**를 그래프 순회로 한 번에 추적할 수 있다. |
| **MySQL** | 화별 요약(`episode_summaries`)만 | 인덱싱 시 화마다 생성해 두는 3~5문장 요약. 표 형태라 굳이 그래프/벡터일 필요가 없어 남겨둠. |

원래는 Neo4j 없이 MySQL의 `entities`/`facts` 테이블(주어-서술어-목적어 + 화 번호)로
그래프 데이터를 흉내 냈다. 이후 실제 Neo4j를 추가하면서, **이미 인덱싱된 MySQL
entities/facts를 LLM 재호출 없이 그대로 그래프로 옮겼다** (`src/graph_migrate.py`) —
추가 비용 $0.

### Neo4j 스키마

```
(Character|Item|Location|Faction|Skill 노드)
        │
   [:HAS_FACT]
        ▼
     (Fact 노드: predicate, episode, valid_from/until_episode, note, object_text)
        │
    [:ABOUT]  (목적어가 엔티티일 때만 존재)
        ▼
(Character|Item|Location|Faction|Skill 노드)
```

관계를 "타입"이 아니라 **노드**(`Fact`)로 승격시킨 이유: `predicate`가 자유 서술어라
`OWNS`/`GAVE`처럼 Neo4j 관계 타입으로 고정하면 동의어(`gave`/`owned_by`/`transferred_to`)
문제로 검색이 새기 쉽다. `Fact` 노드에 속성으로 넣고 이름 필터는 `HAS_FACT`/`ABOUT` 두
관계만으로 양방향(주어든 목적어든) 조회하게 만들어, 실제로 "천인호가 누구 손에
죽었나"(목적어 방향) 같은 질문에서 이전 버전에 있던 방향성 버그를 원천 차단했다.

## 3. 인덱싱 파이프라인

```
episode_N.txt
     │
     ▼  src/indexer.py — chunk_text() (문단 경계 존중, ~1200자 그리디 묶음, 오버랩 없음)
     │
     ▼  src/llm_extractor.py — 화당 LLM 호출 1회
     │    입력: 원문 전체 + "지금까지 알려진 엔티티 이름 목록"(누적 컨텍스트 전체 아님!)
     │    출력: (a) 청크 임베딩 대상 텍스트  (b) entities/facts JSON  (c) 화 요약 3~5문장
     │
     ├─▶ Qdrant: 청크 임베딩 upsert
     ├─▶ MySQL: entities/facts insert, episode_summaries upsert
     └─▶ (별도 실행) src/graph_migrate.py: MySQL entities/facts → Neo4j
```

**비용 설계 포인트**: 이전 화 원문이나 그래프 전체를 매번 프롬프트에 다시 넣지 않는다.
알려진 엔티티 **이름 목록만** 넘겨서 "이 이름이 이미 있으면 재사용, 없으면 새로 생성"
판단을 LLM에 맡긴다. 화 6개 전체 인덱싱 비용 약 $1 수준.

## 4. 에이전트 동작 방식 (LangGraph)

```
        START
          │
          ▼
    ┌──[agent]◀────────────┐
    │     │                │
    │  tool_calls?         │
    │   ├─ 있음, 라운드<한도 → [tools] → [grade] ─┘ (재검색 힌트를 메시지로 추가)
    │   ├─ 있음, 라운드≥한도 → [final_answer] → END   (도구 unbind, 강제 종료)
    │   └─ 없음 → END
```

- **`agent` 노드**: 시스템 프롬프트(+화별 요약, 아래 6번) 기준으로 `vector_search`/
  `graph_query` 중 필요한 걸 골라 호출하거나, 이미 충분하면 바로 답한다.
- **`tools` 노드**: 실제 도구 실행 (`ToolNode`).
- **`grade` 노드**: 방금 결과가 질문에 답하기 충분한지 별도 LLM 호출로 판정
  (Self-RAG 스타일). 불충분하면 "다음엔 이렇게 검색해봐" 힌트를 대화에 얹고 `agent`로
  되돌아간다.
- **`final_answer` 노드**: `MAX_TOOL_ROUNDS`(기본 3) 도달 시 강제 진입. **tools를
  bind하지 않은 LLM**을 쓰기 때문에 이 노드는 물리적으로 tool_calls를 만들 수 없다 —
  "잘 종료해줘"라고 프롬프트로 부탁하는 게 아니라 구조적으로 루프를 끊는다.
  (`recursion_limit=15`도 이중 안전장치로 걸려 있다.)
- **멀티턴**: 별도 rewrite 노드 없음. 시스템 프롬프트에 "지시어(이 사람/그 물건)는
  대화 기록에서 스스로 해석하라"는 지침만 주고, `MemorySaver`(thread_id 키)가 전체
  대화 이력을 유지해 LLM이 문맥으로 해소한다.

### 실제 예시 (Q5 로그, 2026-07-16 검증)

```
[round 0] agent -> graph_query 호출 결정
graph_query 생성 Cypher: 이름이 주어든 목적어든 걸리도록 UNION으로 양방향 조회
graph_query 결과 | 10행
[round 1] grade -> 충분함
[round 1] agent -> 최종 답변 (도구 호출 없음): "정희원 — killed — 천인호 (6화)"
```

## 5. 화별 요약 프리로드 (2026-07-16 추가)

**문제**: 인덱싱 범위(1~6화)를 벗어나는 전역 질문("전체 줄거리 요약해줘", "결말이 뭐야")을
받으면, `vector_search`/`graph_query` 둘 다 "관련 조각"만 찾는 도구라 애초에 그런 정보가
없다. `grade`는 이걸 "검색이 부족해서"라고 오판해 재검색을 반복 요구했고, 라운드 한도까지
소진한 에이전트가 결국 **자기 사전학습 지식**(진짜 완결작 내용)으로 답을 지어내는 걸
실제로 관측했다.

**해결**: 인덱싱 시 화마다 이미 만들어 두던 요약(`episode_summaries`, 그동안 아무 도구도
안 쓰던 죽은 데이터였음)을 매 요청마다 시스템 프롬프트에 통째로 주입한다
(`src/agent.py::_episode_summaries_block`). 화 6개 요약 합쳐도 토큰 비용은 무시할
수준이고, "이 DB는 N~M화까지만 다룬다"는 범위 명시를 함께 넣어 그 밖의 내용은 사전지식이
아니라 "확인 불가"로 답하게 강제한다. `grade` 프롬프트에도 "범위 밖 질문은 재검색 요구하지
말고 바로 정보 없음으로 판단하라"는 지침을 추가했다.

**효과** (실측): "소설 전체 줄거리를 요약해줘"가 이제 **도구 호출 0번, round 0에서 즉시**
1~6화 범위 내에서만 정확하게 답한다 — 정확도와 비용(라운드 3회 소모 → 0회) 둘 다 개선.
"이 소설은 어떻게 끝나?" 같은 범위 밖 질문도 사전지식으로 채우지 않고 정직하게 거절한다.

## 6. 기능 현황

**구현됨**
- 자유 형식 질의응답 (서사/구조화 사실 모두)
- 부정 케이스 처리 (없는 사실을 지어내지 않고 "없음"이라 답함)
- 멀티턴 지시어 해소
- 다단계 연쇄 추적 (아이템이 여러 인물을 거친 이동 경로 등, 그래프 순회)
- 전역 요약 질문 (인덱싱 범위 내에서 근거 있게, 범위 밖은 정직하게 거절)
- 무한 루프/폭주 방지 (라운드 하드캡 + 강제 종료 노드 + recursion_limit)
- 관측성: 라운드별 tool 호출 인자, 생성된 Cypher/SQL, grade 판정을 `logs/agent.log`에 기록
- 간단한 채팅 프론트 (FastAPI + 정적 HTML, thread_id로 세션 유지)

**미구현** (readme.md 원 비전 대비)
- 에디터 통합, 인덱싱 버튼(현재는 CLI로 수동 실행)
- 증분 인덱싱 / 레트콘(설정 변경) 대응
- "다음에 뭘 쓸지 추천" 기능
- 설정오류(모순) 탐지 — 대신 이를 테스트하기 위한 평가 데이터셋(`eval/contradiction_test_set.json`, 15케이스)만 준비됨
- 인증/멀티테넌시, 대화 상태 영속화 (`MemorySaver`는 인메모리라 재시작 시 소실)

## 7. 비용 특성

- 인덱싱(1회성): 6화 전체 약 $1
- 질문 1건: 라운드 수·도구 조합에 따라 대략 $0.03(1라운드, 구조화 조회 위주) ~
  $0.15~0.30(2~3라운드 + 대량 벡터검색)
- 전역 요약처럼 화별 요약으로 즉답 가능한 질문은 이제 도구 호출이 아예 없어 가장 저렴함
- 멀티콜 RAG 방식이 "원문 전체를 매번 컨텍스트에 넣는" 방식보다 토큰 효율이 높고, 이
  격차는 원작이 길어질수록(코퍼스가 커질수록) 구조적으로 더 벌어진다 — 이 방식의 비용은
  질문 난이도에만 비례하고 코퍼스 크기에는 비례하지 않기 때문.

## 8. 알려진 한계 (2026-07-16 기준)

- 에디터 통합, 증분 인덱싱, 레트콘 대응, "다음 작성 추천" 없음 (Q&A 코어만 구현).
- 대화 상태 인메모리 — 서버 재시작 시 소실. 인증/멀티테넌시 없음.
- `structured_query`(MySQL/Text2SQL)는 코드는 남아 있지만 `graph_query`(Neo4j)로
  대체되어 에이전트에는 연결돼 있지 않음.
- 설정오류 탐지는 미구현 (평가 데이터셋만 존재).
