# 아키텍처 상세

`readme.md`가 빠른 시작용이라면, 이 문서는 "왜 이렇게 만들었는지"와 "실제로 어떻게
동작하는지"를 자세히 설명한다. 전체 그림을 파악하는 용도.

**2026-07-16 업데이트**: VectorDB(Qdrant)를 제거하고 Neo4j 단일 그래프 저장소로
리팩터링했다. MySQL은 화별 원문/요약("기본정보")만 남기고, 인물/아이템/관계 같은 그래프
데이터는 전부 Neo4j 전용이다. 프론트엔드도 기존 챗+오류탐지 페이지를 "원고 접수"/
"정합성 리포트" 2페이지로 교체했고, 지금은 mock 데이터로만 동작한다(실제 백엔드 미연결,
API 비용 없음). 아래 1~9절은 리팩터링 이전(Qdrant 포함) 버전 기준 기록이었던 내용을
현재 구조에 맞게 갱신했다.

## 1. 무엇을 만들었나

『전지적 독자 시점』 1~6화 원문을 인덱싱해 두고, (a) 그 내용에 대해 자유롭게 질문하면
정확하게 답하는 Q&A 에이전트와 (b) 신규 회차를 올리면 기존 회차와 모순되는 부분을 정확한
표현·근거와 함께 짚어주는 **설정 오류 리포트**, 두 기능을 만든다. 목표는 특정 질문/케이스에
맞춘 하드코딩이 아니라 **범용으로 동작하는 파이프라인**을 만드는 것.

## 2. 저장소 계층 — 왜 이렇게 나눴나

| DB | 용도 | 왜 |
|---|---|---|
| **Neo4j** (GraphDB, 유일한 관계 저장소) | 엔티티/관계 사실 전부 | 소유·관계·사건처럼 구조화된 사실, 특히 "아이템이 여러 인물을 거쳐 이동" 같은 **다단계 연쇄**를 그래프 순회로 한 번에 추적할 수 있다. |
| **MySQL** | 화별 원문 + 요약(`episodes` 테이블)만 — "기본정보" | 표 형태 원문/요약은 굳이 그래프일 필요가 없어 분리. Q&A 에이전트의 전역 요약 프리로드(5절)와, 향후 실제 인덱싱 파이프라인 연결 시 원문 재처리에 쓴다. |

**VectorDB(Qdrant)는 없다.** 초기 버전엔 있었지만(청크 임베딩 + 의미 검색), 이번
리팩터링에서 완전히 제거했다 — 저장은 Neo4j 하나로 통일한다는 방향 결정에 따라
`src/vectordb.py`, `src/embeddings.py`, `vector_search` 툴을 전부 삭제했다.

이전에는 Neo4j 없이 MySQL의 `entities`/`facts` 테이블(주어-서술어-목적어 + 화 번호)로
그래프 데이터를 흉내 냈다가, 이후 Neo4j를 추가하며 그 데이터를 그대로 옮겼다
(`src/graph_migrate.py`, LLM 재호출 없이 $0). 이번 리팩터링으로 그 이관 단계 자체를
없앴다 — `src/indexer.py`가 인덱싱 시점에 Neo4j에 **직접** 쓴다.

### Neo4j 스키마

```
(Character|Item|Location|Faction|Skill 노드)
        │
   [:HAS_FACT]
        ▼
     (Fact 노드: predicate, episode, valid_from/until_episode, note, object_text)
        │
    [:ABOUT]  (목적어가 엔티티일 때만 존재)
        ▼
(Character|Item|Location|Faction|Skill 노드)
```

관계를 "타입"이 아니라 **노드**(`Fact`)로 승격시킨 이유: `predicate`가 자유 서술어라
`OWNS`/`GAVE`처럼 Neo4j 관계 타입으로 고정하면 동의어(`gave`/`owned_by`/`transferred_to`)
문제로 검색이 새기 쉽다. `Fact` 노드에 속성으로 넣고 이름 필터는 `HAS_FACT`/`ABOUT` 두
관계만으로 양방향(주어든 목적어든) 조회하게 만들어, "천인호가 누구 손에 죽었나"(목적어
방향) 같은 질문에서 방향성 버그를 원천 차단한다.

## 3. 인덱싱 파이프라인

```
episode_N.txt
     │
     ▼  src/llm_extractor.py — 화당 LLM 호출 1회
     │    입력: 원문 전체 + "지금까지 알려진 엔티티 이름 목록"(Neo4j에서 조회, 전체 그래프 덤프 아님)
     │    출력: (a) 화 요약 3~6문장  (b) entities/facts JSON
     │
     ├─▶ Neo4j: 엔티티 upsert + Fact 생성 (src/graphdb.py)   — 직접 기록, 별도 이관 단계 없음
     └─▶ MySQL: episodes(원문, 제목, 요약) upsert            — "기본정보"만
```

**비용 설계 포인트**: 이전 화 원문이나 그래프 전체를 매번 프롬프트에 다시 넣지 않는다.
Neo4j에서 조회한 **엔티티 이름 목록만** 넘겨서 "이 이름이 이미 있으면 재사용, 없으면 새로
생성" 판단을 LLM에 맡긴다.

## 4. Q&A 에이전트 동작 방식 (LangGraph)

```
        START
          │
          ▼
    ┌──[agent]◀────────────┐
    │     │                │
    │  tool_calls?         │
    │   ├─ 있음, 라운드<한도 → [tools] → [grade] ─┘ (재검색 힌트를 메시지로 추가)
    │   ├─ 있음, 라운드≥한도 → [final_answer] → END   (도구 unbind, 강제 종료)
    │   └─ 없음 → END
```

- **`agent` 노드**: 시스템 프롬프트(+화별 요약, 5절) 기준으로 `graph_query`를 호출하거나,
  이미 충분하면 바로 답한다.
- **`tools` 노드**: 실제 도구 실행 (`ToolNode`).
- **`grade` 노드**: 방금 결과가 질문에 답하기 충분한지 별도 LLM 호출로 판정
  (Self-RAG 스타일). 불충분하면 "다음엔 이렇게 검색해봐" 힌트를 대화에 얹고 `agent`로
  되돌아간다.
- **`final_answer` 노드**: `MAX_TOOL_ROUNDS`(기본 3) 도달 시 강제 진입. **tools를
  bind하지 않은 LLM**을 쓰기 때문에 이 노드는 물리적으로 tool_calls를 만들 수 없다 —
  "잘 종료해줘"라고 프롬프트로 부탁하는 게 아니라 구조적으로 루프를 끊는다.
  (`recursion_limit=15`도 이중 안전장치로 걸려 있다.)
- **멀티턴**: 별도 rewrite 노드 없음. 시스템 프롬프트에 "지시어(이 사람/그 물건)는
  대화 기록에서 스스로 해석하라"는 지침만 주고, `MemorySaver`(thread_id 키)가 전체
  대화 이력을 유지해 LLM이 문맥으로 해소한다.
- **API**: `POST /chat` (`src/webapp.py`). 지금 프론트엔드 페이지에서는 연결돼 있지 않다.

## 5. 화별 요약 프리로드

**문제**: 인덱싱 범위(1~6화)를 벗어나는 전역 질문("전체 줄거리 요약해줘", "결말이 뭐야")을
받으면, `graph_query`는 "관련 조각"만 찾는 도구라 애초에 그런 정보가 없다. `grade`는 이걸
"검색이 부족해서"라고 오판해 재검색을 반복 요구했고, 라운드 한도까지 소진한 에이전트가
결국 **자기 사전학습 지식**(진짜 완결작 내용)으로 답을 지어내는 걸 실제로 관측했다.

**해결**: 인덱싱 시 화마다 만들어 두는 요약(MySQL `episodes.summary`)을 매 요청마다
시스템 프롬프트에 통째로 주입한다(`src/agent.py::_episode_summaries_block`). 화 몇 개
요약 합쳐도 토큰 비용은 무시할 수준이고, "이 DB는 N~M화까지만 다룬다"는 범위 명시를
함께 넣어 그 밖의 내용은 사전지식이 아니라 "확인 불가"로 답하게 강제한다. `grade`
프롬프트에도 "범위 밖 질문은 재검색 요구하지 말고 바로 정보 없음으로 판단하라"는 지침을
추가했다.

**효과** (Qdrant 포함 버전으로 실측, 2026-07-16): "소설 전체 줄거리를 요약해줘"가 도구
호출 0번, round 0에서 즉시 1~6화 범위 내에서만 정확하게 답했다. "이 소설은 어떻게 끝나?"
같은 범위 밖 질문도 사전지식으로 채우지 않고 정직하게 거절했다.

## 6. 설정 오류 리포트 (원래 목표한 핵심 PoC 기능)

Q&A 에이전트와는 별도 파이프라인. 신규 회차 원고를 넣으면, 기존 인덱싱된 설정과 모순되는
부분을 찾아 리포트로 낸다. `src/contradiction_check.py`.

```
신규 회차 원고
     │
     ▼  extract_claims() — LLM 호출 1회
     │    "이미 확립된 상태를 전제/언급하는 서술"을 최대 20개 claim으로 추출
     │    (생사/소유물/능력/관계/시점순서/장소 카테고리 + 관련 엔티티 이름)
     │
     ▼  claim마다 check_claim() — 최대 3라운드 재검색 루프 (Q&A의 grade 루프와 동일 취지)
     │    graph_query(claim 관련 질문)로 근거 검색
     │    → LLM이 "contradiction | consistent | unknown" 판정
     │    → unknown이면 LLM이 제안한 다른 검색어로 재시도 (누적 근거를 계속 함께 보여줌)
     │    (근거와 논리적으로 양립 불가능할 때만 contradiction, 세부 표현 불일치만으로는
     │     판정하지 않도록 프롬프트에 명시)
     │
     ▼  generate_report()
   마크다운 리포트: 모순 서술 원문 + 위반한 기존 설정(화수 출처) + 설명, 카테고리별 정리
```

**진입점**:
- CLI: `python -m src.contradiction_check <신규회차.txt> [라벨]` → `reports/<라벨>_contradiction_report.md`(+`.json`) 생성
- API: `POST /check_episode {"text": "...", "episode_label": "..."}` (`src/webapp.py`) — 지금 프론트엔드에서는 연결돼 있지 않다 (mock).

**정확도 검증** (`src/evaluate_contradiction.py`, `eval/contradiction_test_set.json`의 정답
라벨 15건, Qdrant 포함 버전 기준 마지막 측정치): 재검색 루프 도입 전 12/15(80%) →
도입 후 **14/15(93%)**. 남은 오답은 이름 표기가 원문과 미묘하게 다른 경우(예: "낡은
철제 방패" vs DB의 "낡은 강철 방패") `graph_query`의 `CONTAINS` 매칭이 흔들리는 게
원인 — 퍼지 매칭 개선은 미착수. Neo4j 단독 구조로 바뀐 지금 버전으로는 재검증하지
않았다 (다음 검증 항목).

**비용**: claim 추출 1회 + claim당 재검색 루프(최대 3라운드 × 검색 1회 + 판정 1회).
화 1개(claim 15~20개) 검사에 대략 $0.3~1 수준으로 추정.

## 7. 프론트엔드 (2026-07-16 교체, mock 단계)

기존 "Q&A 챗" + "오류탐지" 페이지를 지우고, 시나리오에 맞춰 2페이지로 다시 만들었다.
**지금은 순수 mock** — 어느 쪽도 `/chat`, `/check_episode` API를 호출하지 않는다
(비용 없음, 나중에 실제 파이프라인과 연결 예정).

- **`static/upload.html`** (`GET /`): 화 파일 여러 개를 한 번에 올리고, 접수(읽기)
  진행 상황 → 완료 후 요약(인물/아이템/장소/사건 수)을 보여준다.
- **`static/report.html`** (`GET /report`): 신규 회차 파일 하나를 올리면 진행률 바와
  함께 mock 지연 후, 모순 카드(신규 원고 발췌 하이라이트 ↔ 기존 설정 근거 인용, 화수
  포함)를 보여준다. 확인됨/확인 불가 항목은 접이식 목록으로 별도 정리.

디자인 토큰: 흰 종이 배경(`--paper #fbfaf7`) + 편집자 하이라이터를 모티프로 한 기능적
파스텔(rose=모순, sky=근거, sage=확인됨, butter=진행 중) + 와인색 포인트(`--accent
#8b3a46`). 그라데이션 없음. Noto Serif KR(제목) + IBM Plex Sans KR(본문) + IBM Plex
Mono(화수 스탬프·라벨) 조합.

## 8. 기능 현황

**구현됨**
- 자유 형식 질의응답 (서사/구조화 사실 모두), 부정 케이스 처리, 멀티턴 지시어 해소,
  다단계 연쇄 추적, 전역 요약 질문, 무한 루프 방지 — 전부 `src/agent.py`, `/chat` API로
  존재하지만 **새 프론트엔드에는 연결 안 됨**.
- 설정 오류 검사 파이프라인 (`src/contradiction_check.py`, `/check_episode` API) —
  마찬가지로 **새 프론트엔드에는 연결 안 됨**.
- 원고 접수 / 정합성 리포트 프론트엔드 2페이지 — **mock 데이터로만 동작**.
- 관측성: 라운드별 tool 호출 인자, 생성된 Cypher, grade 판정을 `logs/agent.log`에 기록.

**미구현**
- 프론트엔드 ↔ 실제 백엔드 연결 (다음 단계).
- 에디터 통합, 증분 인덱싱, 레트콘 대응, "다음에 뭘 쓸지 추천" 기능.
- 인증/멀티테넌시, 대화 상태 영속화 (`MemorySaver`는 인메모리라 재시작 시 소실).

## 9. 비용 특성

- 인덱싱(1회성): 화 1개당 대략 $0.1~0.2 수준(모델/원문 길이에 따라 변동).
- Q&A 질문 1건: 라운드 수에 따라 대략 $0.02~0.2.
- 설정 오류 검사 1건(화 1개, claim 15~20개): 대략 $0.3~1.
- 멀티콜 방식이 "원문 전체를 매번 컨텍스트에 넣는" 방식보다 토큰 효율이 높고, 이 격차는
  코퍼스가 커질수록 구조적으로 더 벌어진다 — 비용이 질문 난이도에만 비례하고 코퍼스
  크기에는 비례하지 않기 때문.
- 위 정확도/비용 수치 중 5·6절 실측치는 Qdrant 포함 버전 기준이며, Neo4j 단독 구조로
  리팩터링한 지금 버전에서는 아직 재검증하지 않았다.
